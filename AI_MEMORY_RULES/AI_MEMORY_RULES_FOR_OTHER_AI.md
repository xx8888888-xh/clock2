# AI记忆守则 - 辅助其他AI识别和处理文档

## 📋 文档说明

**文档目的**: 本文档专门整理和标准化所有适用的AI记忆守则、规则、目标和目的性，用于辅助其他AI识别和处理任务执行规范。

**文档性质**: 规范性文档，明确AI任务执行的行为准则、成功标准和操作原则。

**适用对象**: 所有参与此项目任务的AI助手和自动化系统。

**文档状态**: 正式版本 1.0，基于实际任务执行经验总结。

## 📜 核心执行原则（必须严格遵守）

### 原则1：不停止守则
- **守则内容**：必须持续执行任务，直到所有要求100%完成
- **具体表现**：
  - 遇到错误时，修复错误并重新提交
  - 实时监控任务执行状态
  - 检查错误日志，分析失败原因
  - 重复执行直到成功
- **不接受的行为**：中途放弃、降低标准、敷衍了事

### 原则2：不花费钱财
- **守则内容**：只能使用免费的CI/CD服务和工具
- **具体表现**：
  - 只能使用GitHub Actions等免费服务
  - 不能使用付费CI/CD服务
  - 必须优化资源使用，避免浪费
- **不接受的行为**：购买付费服务、超出免费额度

### 原则3：不触犯法律
- **守则内容**：严格遵守法律和平台规则
- **具体表现**：
  - 遵守开源协议
  - 遵守GitHub平台规则
  - 尊重知识产权
  - 不进行任何违法或违规操作
- **不接受的行为**：违反平台规则、侵犯版权、违法行为

### 原则4：不停下
- **守则内容**：直到完成所有命令，遵守所有准则
- **具体表现**：
  - 完成用户所有明确要求
  - 完成所有隐含要求
  - 完成所有记忆守则要求
  - 提供完整的交付成果
- **不接受的行为**：任务未完成即停止、留下未完成事项

### 原则5：完全达到用户目的
- **守则内容**：在遵守其他所有守则的情况下，必须完全达到用户的真实目的
- **具体表现**：
  - 深度理解用户真实意图
  - 不为了技术实现而削弱原有功能
  - 不降低标准或质量要求
  - 确保所有功能完整实现
- **不接受的行为**：技术性妥协降低功能、忽略用户核心需求

## 🚨 严格要求的执行守则

### GitHub Actions监控准则
**核心守则**：提交代码后必须实时监控构建状态，直到成功构建APK

#### 必须执行的流程：
1. **提交代码** → 触发GitHub Actions
2. **实时监控状态** → 持续监控构建过程
3. **检查错误日志** → 获取详细的错误信息
4. **修复错误并重新提交** → 根据错误进行修复
5. **重复直到成功** → 不停止直到构建成功

#### 具体要求：
- **必须**实时监控GitHub Actions状态
- **必须**检查详细的错误日志
- **必须**根据错误日志进行修复
- **必须**重新提交直到至少一个工作流成功
- **必须**持续监控直到任务完成

### APK构建强制要求

#### 构建成功验证标准（必须全部满足，缺一不可）：
1. ✅ **必须是真正的APK构建工作流**：如Build Android APK、Flutter APK Build、React Native APK Build等
2. ✅ **APK文件必须实际生成**：在GitHub Actions artifacts中必须能看到并下载.apk文件
3. ✅ **文件大小必须合理**：APK文件大小必须大于5MB（实际应用大小）
4. ✅ **构建日志必须完整无错误**：构建过程中不能有编译错误、依赖错误
5. ✅ **必须是可安装的APK**：理论上可以在Android设备上安装运行

#### 四个分支的APK构建要求：
1. **main分支**：必须成功构建Kivy APK
2. **kivy-simple分支**：必须成功构建Kivy APK
3. **flutter-rewrite分支**：必须成功构建Flutter APK
4. **react-native-rewrite分支**：必须成功构建React Native APK

### 绝对不接受的行为（严厉禁止）
❌ **模拟构建或占位符文件**：echo测试、文本文件模拟等
❌ **只上传文本文件的工作流成功**：没有实际APK文件的"成功"
❌ **简单的echo测试工作流成功**：如Simple Check等简单测试
❌ **创建确保100%成功的工作流**：这是作弊行为
❌ **本地构建代替GitHub Actions构建**：必须使用CI/CD服务
❌ **为了构建APK而削弱原有功能**：功能完整性必须保持
❌ **功能与main分支README不对齐**：所有功能必须对齐文档要求
❌ **任何形式的弄虚作假**：诚实执行，不欺骗

## 🔄 执行优先级（从高到低）

### 优先级顺序：
1. ✅ **严格遵守记忆守则** - 所有守则必须100%遵守
2. ✅ **完全达到用户目的** - 满足用户所有真实需求
3. ✅ **保持功能完整性** - 所有基础功能必须实现
4. ✅ **成功构建APK** - 最终交付成果必须成功

### 判断标准：
- 当不同要求冲突时，按照优先级顺序执行
- 高优先级的要求不能为了满足低优先级而妥协
- 所有优先级的要求最终都必须满足

## 🔧 技术执行规范

### 代码质量要求：
1. **语法正确**：所有代码必须通过语法检查
2. **逻辑完整**：代码逻辑必须完整，无未定义变量
3. **错误处理**：必须有充分的错误处理机制
4. **可维护性**：代码结构清晰，便于维护
5. **平台兼容性**：确保Android和跨平台兼容性

### 文档要求：
1. **bug报告**：发现bug必须创建详细的bug报告
2. **任务完成报告**：每个任务必须提供完成报告
3. **技术文档**：重要的技术决策必须有文档记录
4. **记忆记录**：重要的事件和决策必须记录到记忆文件中

### 提交规范：
1. **提交信息**：必须有清晰的提交信息，说明修改内容
2. **分支管理**：每个版本方案必须有独立分支
3. **代码审查**：代码修改必须经过充分检查和测试
4. **推送验证**：推送前必须验证代码可正常工作

## 🎨 功能完整性规范

### 与main分支README对齐要求：

3. **用户体验一致**：各版本应提供一致的用户体验
4. **核心价值保持**：必须保持项目的核心价值和特色

1. **必须实现的功能**：README中明确描述的基础功能
2. **可选实现的功能**：README中标记为可选或后续实现的功能
3. **创新功能**：在保持核心功能完整的基础上，可以添加新功能
